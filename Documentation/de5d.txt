- CQ: Inconsistent code style/patterns. For example, sometimes a function `isValidFloor` is used to check if a floor is valid, sometimes an if is inlined. Etc.
- CQ: Some functions are very long and contain several checks and assignments to local variables before doing what they are supposed to do. An example is `publishToConsumers`. This function could benefit from refactoring into smaller functions with dedicated responsibilities.
- CQ: Some functions contain nested ifs inside of loops inside of switches inside of loops. To improve readability of control flow, this should be refactored with a focus on less nesting.
- CQ: Magic numbers in channel initialization in `main.go`, explained to avoid deadlocks, but not the choice of the actual numbers.
- State: Generally, all module states are private and communicated through others through channels. The main entry point makes it clear which modules exists and the channels they communicate on.
- Coupling: The boundary between `callhandler` and `controller` is not immediately clear. In particular, it is hard to quickly get an overview over all the cases where the physical elevator control outputs can change.
- Coherence: `callhandler` does a lot of things, and could benefit from being split into either several functions with defined responsibilities, or separate modules.
Overall this code had a pretty good structure, but seemed a bit unpolished code quality-wise. Control flow was hard to read. The biggest plus of the code base was state management through message passing, which strengthened encapsulation and coherence.
Since the main drawbacks of the code are localized to function bodies / blocks of code, we believe these issues are less severe.
